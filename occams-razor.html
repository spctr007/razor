01 / Foundation

What is
Occam's Razor?

Occam's Razor is a problem-solving principle stating that among competing hypotheses, the one with the fewest assumptions should be selected. In software, it means: don't build more than the problem requires.

Entities must not be multiplied beyond necessity. When two explanations fit the facts equally well, prefer the simpler one.

— William of Ockham, 14th century philosopher — applied daily by engineers who ship


In software, this translates directly: don't introduce abstractions, patterns, or infrastructure that the current problem doesn't require. Every layer you add must earn its place — it carries a cost in complexity, maintenance, and cognitive load for every engineer who comes after you.


With AI tools, this matters more than ever — because AI will happily generate sophisticated, pattern-heavy, over-engineered solutions to simple problems if you don't ask for simpler ones explicitly.

02 / Core Framework

The 5 Principles
Applied to Software

Click each principle to expand it with software engineering examples and how AI fits in.

PRINCIPLE 01
Prefer the Simpler Explanation
Don't multiply abstractions beyond what the problem demands.

Every architectural element — a message queue, an event bus, a caching layer, a service mesh — exists to solve a real problem. If you can't articulate which specific problem it solves for you right now, you probably don't need it. The cost of unnecessary complexity is paid by every engineer who reads your code afterward.

Software Example

"We need Kafka for our notification system." Why? "For scalability." Current load: 50 events/day. A simple database table with a background job does the same job with 90% less operational complexity and zero new infrastructure to learn.

AI Application

When AI suggests an architecture, ask: "What is the simplest solution that meets only the stated requirements? Give me that first, then tell me what would need to change if scale increased 100x."

PRINCIPLE 02
Measure Cognitive Load, Not Line Count
Complexity lives in the reader's head, not the editor's line counter.

Simple code isn't necessarily short code. A 10-line function that introduces 4 new concepts is more complex than a 40-line function that reads like plain English. Occam's Razor applied to code means minimizing the number of concepts, not characters. Fewer moving parts, fewer mental models to hold.

Software Example

A clever one-liner using reduce, flatMap, and a custom comparator replaces 20 lines of a readable loop. The one-liner "wins" on line count and loses on every other metric: it's harder to debug, harder to modify, harder to read in a 2am incident.

AI Application

Tell AI: "Write this for a junior engineer joining the team tomorrow, not for impressing senior engineers. Optimize for clarity, not cleverness. If you use an advanced language feature, explain whether a simpler form would work just as well."

PRINCIPLE 03
Eliminate Unnecessary Layers
Every abstraction has a cost. Make each layer earn its place.

Abstraction is a tool, not a virtue. A wrapper class that wraps a wrapper that wraps an interface that wraps a library call doesn't make the code cleaner — it makes it harder to trace, harder to debug, and harder to delete. Good software has the minimum number of layers needed to separate genuinely different concerns.

Software Example

A "Repository pattern" over an ORM over a database driver is three layers to query one table. Ask: does each layer represent a genuinely separate concern, or is it abstraction for abstraction's sake? If you collapsed two of them, would anything get harder to change independently?

AI Application

Ask AI: "Here's my proposed layering. For each layer, tell me: what specifically would be harder to change if this layer didn't exist? If the answer is 'nothing', let's remove it."

PRINCIPLE 04
Don't Architect for Hypotheticals
Solve the problem you have. The future will bring its own requirements.

Over-engineering is almost always driven by imagined futures: "we might need multi-tenancy," "we might need to swap databases," "we might need to scale to millions of users." These futures rarely arrive as imagined — and when they do, the pre-built abstractions are usually wrong anyway. Build for today's requirements with today's constraints. Refactor when the future becomes the present.

Software Example

An API designed to support "any payment provider" through an abstraction layer, when you have exactly one payment provider and no plans to change. The abstraction makes Stripe harder to use while providing no current value. YAGNI: You Aren't Gonna Need It.

AI Application

Before asking AI to make something "extensible" or "pluggable" — stop. Ask instead: "What are the actual requirements I need to meet this week? Build exactly that, and tell me what would need to change if requirement X appeared in 6 months."

PRINCIPLE 05
Simple ≠ Simplistic
Simplicity requires more thought, not less. Earned simplicity is the hardest kind.

Occam's Razor is not an excuse for lazy design. A "simple" solution that ignores error handling, security, or edge cases isn't simple — it's incomplete. True simplicity means a solution that is complete, correct, and easy to understand. It often takes more effort to produce than a complex one, because you have to ruthlessly cut without losing correctness.

Software Example

A simple REST endpoint that is also safe (validates input, returns correct error codes, handles concurrency, has a timeout) is harder to write than one that just works for the happy path. Simplicity in design — few moving parts — is different from simplicity in effort.

AI Application

Ask AI: "Is this solution actually simple, or is it just incomplete? What necessary complexity am I dropping by going with this approach? What's the minimum I need to add to make this production-safe without adding unnecessary layers?"

03 / Practical Method

The Occam's Razor
AI Workflow

A five-step method for applying the razor before, during, and after generating code with AI.

STEP 01 ✂️
State Only What You Need Now

Before prompting, write down your actual requirements — not imagined future ones. Strip qualifiers like "someday," "maybe," and "in case we." Only what you need to ship this week counts.

"My current requirements are:
[list only concrete, present-tense requirements]

Explicitly NOT in scope: [list things that sound tempting but aren't needed yet]

Design the simplest solution that meets only the in-scope requirements.
Do not add flexibility, extensibility, or abstraction beyond what is needed."
STEP 02 🔬
Challenge Every Component

For every piece of the proposed solution, ask: what specific requirement does this serve? If you can't answer, it's a candidate for removal.

"Here's the solution you proposed:
[paste solution]

For each component or layer, answer: what specific requirement
from my list does this serve?

Flag anything that doesn't map directly to a stated requirement.
Then suggest what to remove."
STEP 03 📉
Ask for the Simpler Version

After seeing the first solution, always ask for a simpler one. AI defaults to comprehensive. You want minimal.

"Now give me a version that is meaningfully simpler.
Drop any pattern or layer that isn't strictly required.

What are the concrete trade-offs of the simpler version
versus the previous one?
Under what conditions would the simpler version break?"
STEP 04 ⚖️
Evaluate What You'd Lose

For each thing removed in simplification, understand the cost. Some complexity is essential — removing it is not simplicity, it's incompleteness. Know the difference before deciding.

"Between the simpler and more complex versions,
what does the simpler one handle incorrectly or not at all?

Separate this into two lists:
1. Things I'd genuinely need to add back (edge cases, errors, security)
2. Things I only thought I'd need (premature flexibility, speculative scale)"
STEP 05 🚀
Ship and Set a Complexity Budget

Once you've applied the razor, ship the result. Set a written rule: no new abstraction layers without a concrete requirement that currently can't be met without them.

"We've shipped the simple version. Help me write a one-paragraph
decision record that captures:
1. What we chose and why
2. What we explicitly decided NOT to build
3. What specific signal would tell us to add complexity later

This will live in our docs so future engineers don't re-add
what we deliberately removed."
04 / Anti-Patterns

When Engineers
Violate the Razor

These are the most common ways complexity gets added without necessity — and what applying the razor looks like instead.

Scenario
Over-engineered
Occam's Approach
User authentication for a 3-person internal tool
OAuth2 with identity provider, refresh token rotation, role-based permission matrix, audit logging service
HTTP Basic Auth or a shared secret header. Add OAuth when the team grows or compliance requires it.
Config values that differ between environments
Feature flag system with a remote dashboard, SDK integration, percentage rollouts, A/B experiment tracking
Environment variables. A `.env` file. Change the value to change the behavior.
Storing 500 records that rarely change
Redis cache in front of PostgreSQL with cache invalidation events, TTL tuning, warm-up scripts
Query the database. Add a cache when profiling shows the query is actually slow.
Background job that runs once a day
Distributed task queue (Celery/BullMQ), worker autoscaling, retry policies, dead-letter queue, monitoring dashboard
A cron job. Or a simple database row with a `run_at` column checked every minute.
API consumed by a single frontend you own
Versioned REST API with hypermedia links, OpenAPI spec, SDK generation, backwards-compatibility policy
Unversioned endpoints. Change both sides together. Add versioning when a second consumer appears.
Sharing code between two services
Private npm/PyPI package with its own repo, CI/CD, semver, changelog, and dependency update bot
Copy the code. Refactor to a shared package when the duplication becomes an actual maintenance problem.
05 / Worked Example

The Notification
System

A team is asked to build a notification system for their SaaS product. Watch how Occam's Razor changes the outcome.

⚡ Case Study
"We need to notify users when things happen in the system."

Actual requirements gathered: send an email when a report is ready. ~200 users. Reports run nightly.

THE OVER-ENGINEERED RESPONSE

→
Event Bus Kafka cluster to emit domain events from every service. "Notifications are just one consumer — this will scale to anything."
→
Notification Service Dedicated microservice with its own database, deployment pipeline, and API. Subscribes to the event bus.
→
Template Engine Handlebars-based template system with a database-backed template store, so non-engineers can edit emails.
→
Preference Center User-facing notification settings UI. Per-event type toggles. Stored in the notification service DB.

Timeline: 6 weeks. Three engineers. Still not shipped.

THE OCCAM'S RAZOR RESPONSE

→
A cron job Runs at 6am. Queries: "which reports finished since yesterday?" One SQL query.
→
Sendgrid API call For each result row, send one email using a hardcoded template. Plain text with one link.
→
A log line Print "Sent N emails" to stdout. Your existing log aggregator captures it.

Timeline: 1 day. One engineer. Ships before standup.

What Changes When Requirements Change?

If users need real-time notifications → add WebSockets at that point. If templates need to be editable → add a template table then. If volume hits 50,000 users → profile first, optimize second. Each requirement, when it actually arrives, drives the right addition. No premature complexity needed.

06 / Test Yourself

Apply the Razor

Three scenarios. Apply Occam's Razor and see if your instinct aligns with the principle.

A startup with 50 users wants to "future-proof" their data layer by building a custom ORM abstraction over PostgreSQL so they can "swap databases later." What does Occam's Razor say?
Correct. Database swaps are rare, and when they happen, the migration is never clean regardless of your abstraction layer. The simpler explanation: use your ORM directly, solve the problem you have, and revisit if the hypothetical ever materializes. YAGNI and Occam's Razor agree.
Which of these is an example of violating Occam's Razor, rather than applying it?
Correct. A plugin architecture multiplies entities (abstractions, interfaces, conventions) beyond necessity when there's only one use case and one team. Options A skips necessary complexity (error handling is required, not optional). Options B and D are good Razor applications — simpler solutions for simpler problems.
A team ships a "simple" authentication system with no input validation, no rate limiting, and no token expiry. They justify it as "Occam's Razor — fewest moving parts." Are they applying the principle correctly?
Correct. Occam's Razor eliminates unnecessary complexity, not necessary complexity. Input validation, rate limiting, and token expiry are requirements of a correct auth system — not optional extras. Removing them makes the solution incomplete, not simple. Simple ≠ simplistic.
07 / Quick Reference

The Razor
Cheatsheet

Keep these questions close when you're designing or reviewing code.

✂️
The Razor Question
  • Does this layer exist to solve a real, current requirement?
  • Can I name the specific problem this abstraction solves?
  • What happens if I delete this? Does anything break?
  • Is this complexity earned, or speculative?
🚩
Red Flags
  • "We might need this later"
  • "Let's make this pluggable/extensible"
  • "This is the industry standard pattern"
  • "Let's abstract this so we can swap it out"
  • "This will scale to millions"
✅
Simple ≠ Simplistic
  • Error handling is required, not optional
  • Security is never premature complexity
  • Observability belongs in day one
  • Simplicity means fewer moving parts, not fewer safety nets
🤖
AI Prompts
  • "Give me the simplest version that meets only these requirements"
  • "What would you remove if I told you we have 100 users?"
  • "What's missing from the simpler version that's actually required?"
  • "Write a decision record for what we chose not to build"
📐
The Complexity Budget
  • Every new layer needs a written justification
  • No abstraction without a second use case
  • No caching before profiling
  • No queue before the synchronous version fails
🔄
When to Add Complexity
  • When a specific requirement can't be met without it
  • When profiling shows a concrete bottleneck
  • When a second consumer actually appears
  • When the simpler version demonstrably fails in production
